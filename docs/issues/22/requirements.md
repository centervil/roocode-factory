# 要件定義: Issue #22 - スラッシュコマンドの定義統合とメタポリシーの策定（再定義）

## 1. 背景と目的
現在のスラッシュコマンドは `.gemini/` (TOML) と `.roo/` (Markdown) で別々に管理されており、同一のコマンドであっても定義が重複・乖離しています。また、スラッシュコマンドが開発プロセス全体においてどのような役割（責任範囲）を持つべきかが不明確です。
本 Issue では、これらを統合する「マスター定義」の仕組みを構築し、スラッシュコマンドを IDD プロセスにおける「意図の起動（Intent Trigger）」として再定義します。

## 2. ターゲットユーザー / シナリオ
- **開発者**: 1つのマスターファイルを更新するだけで、Gemini CLI と Roo Code 両方のコマンドを同期させたい。
- **エージェント**: スラッシュコマンドを通じて、どのスキルを使用し、どのフェーズを開始すべきかを明確に認識したい。
- **ユーザー**: ツールを問わず、同じコマンド名で同じ振る舞いを期待したい。

## 3. 深掘りされた要件
### 3.1 技術的統合 (Unified Definition)
- **マスターフォーマットの選定**: Gemini CLI (TOML) と Roo Code (Markdown + YAML Frontmatter) の両方に変換可能なマスター形式（例: TOML または YAML）を定義する。
- **自動同期メカニズム**: マスター定義から `.gemini/commands/` および `.roo/commands/` へ各形式で書き出すスクリプト（または手順）を整備する。
- **引数仕様の統一**: `{{issue-id}}` などのプレースホルダー記法を統一し、両環境で正しく解釈されるようにする。

### 3.2 概念的・構造的定義 (Metapolicy)
- **開発プロセスにおける役割の明確化**:
    - **憲法 (AGENTS.md)**: AI の行動原理、アイデンティティ（「何者であるか」）。
    - **地図 (README.md)**: プロジェクト構造、Source of Truth（「どこに何があるか」）。
    - **マニュアル (SKILL.md)**: 具体的な手順、ツール操作（「どうやるか」）。
    - **意図の起動 (Slash Commands)**: 特定のコンテキストにおける「目的」の宣言（「今、何をしたいか」）。
- **責務の分離**: スラッシュコマンド自体には複雑な手順を書かず、必ず「`activate_skill` して 〇〇 スキルを実行せよ」という「トリガー」に徹する。

## 4. 完了定義 (Definition of Done: DoD)
- [ ] スラッシュコマンドのマスターフォーマット（案）が承認されていること。
- [ ] マスター定義から各環境用ファイルを生成するスクリプト（`sync-commands.sh` 等）が実装されていること。
- [ ] 既存の全コマンドがマスター定義に移行され、重複が排除されていること。
- [ ] `AGENTS.md` または `README.md` に、スラッシュコマンドを「意図の起動 (Intent Trigger)」として定義する記述が追加されていること。

# 要件定義: Issue #22 - スラッシュコマンドの運用方針の策定

## 1. 背景
Roocode/Gemini では、特定の AI エージェントのワークフローを起動するためにスラッシュコマンドを使用しています。しかし、現在、これらのコマンドの定義や使用方法は、異なる環境（.gemini と .roo）間で一貫性がなく、明確な運用方針が欠如しています。プロジェクトが Release Candidate (RC) フェーズに近づくにあたり、安定性と使いやすさを確保するために、正式な運用方針を策定する必要があります。

## 2. ターゲットユーザー / シナリオ
- **開発者**: 新しいコマンドをどのように定義し、どこに配置すべきかについての明確なガイドラインが必要。
- **ユーザー**: システムと対話するための、一貫性があり予測可能なインターフェースが必要。
- **メンテナー**: コマンドを効率的に監査・更新するための標準化された構造が必要。

## 3. 要件
- **命名規則の標準化**: コマンドの明確な命名規則（例：kebab-case、特定の接頭辞など）を定義する。
- **ディレクトリ構造の整理**: `.gemini/commands/` と `.roo/commands/` の役割を明確にする。
- **コマンドスキーマの標準化**: コマンド定義のフォーマット（TOML か Markdown か）を標準化する。
- **スキルとのマッピング**: コマンドを特定のスキルにどのようにマッピングすべきか（例：1スキル1コマンドか、複数か）を定義する。
- **プロンプトエンジニアリング基準**: エージェントの一貫した高品質な振る舞いを保証するために、コマンド定義内の `prompt` セクションの記述ルールを策定する。
- **ドキュメント化**: 利用可能なコマンドをユーザーに提示し、説明するための自動または手動の方法を検討する。

## 4. 完了定義 (Definition of Done: DoD)
- [ ] 正式な運用方針ドキュメント（`docs/policies/slash-commands.md`）が作成され、承認されていること。
- [ ] `.gemini/commands/` および `.roo/commands/` 内の既存のコマンドが新しい方針に沿って監査されていること。
- [ ] 現在のコマンド実装における不整合を特定し、移行プランが提案されていること。
- [ ] 必要に応じて、新しいコマンド標準をサポートするように `skill-project-and-skill-architect` が更新されていること。

# Issue #55 要件定義: RAW ログと信頼性スコアに基づく /self-optimize の刷新

## 1. 背景と目的
従来の `/self-optimize` は AI の自己申告ログ (`self_optimization.json`) に依存していた。
本刷新により、`.ops/audit_logs/raw/` の客観的な実行ログと `integrity_audit.py` による信頼性スコア (Integrity Score) を分析の主軸に据え、証拠に基づいた自己改善サイクルを確立する。

## 2. 業務フロー (To-Be)
1. **データ収集**: `/self-optimize` 実行時、直近 1 セッションの RAW ログと開発ログを特定する。
2. **監査実行**: `integrity_audit.py` を動的に実行し、最新の Integrity Score とエラー内訳を算出する。
3. **分析**: 
    - 低スコアの原因（ツールの誤用、成功の虚偽申告、調査不足等）を RAW ログから特定。
    - 過去のメトリクス (`.ops/audit_logs/metrics.json` 等) と比較し、改善傾向または改悪傾向を把握。
4. **提案**: モード指示書 (`.roo/rules-*`) やプロンプト、スキルの修正案 (Diff) を生成。
5. **実行**: ユーザーの承認を得た上で、ファイルを修正する。

## 3. 機能要件
- **RAW ログ解析機能**: `.ops/audit_logs/raw/` 内の最新ログを自動取得し、ツール実行の成否を客観的に評価する。
- **動的監査統合**: コマンド内で `integrity_audit.py` を呼び出し、その出力を分析インプットとして利用する。
- **改善提案ロジック**: 
    - 信頼性スコアが 80 点未満の場合、具体的な「なぜ失敗したか」に対するルールレベルの対策を提示。
    - 過去のセッションデータとの比較による「振り返り」メッセージの生成。
- **UI/UX**: ユーザーに対し、現在のスコア、分析結果、修正案を構造化して提示する。

## 4. 完了条件 (Success Criteria)
- [ ] `/self-optimize` が実行ログから客観的な Integrity Score を算出し、表示すること。
- [ ] スコア低下の原因に基づいた、具体的かつ妥当な `.roo/rules-*` 等への修正案が提示されること。
- [ ] 過去の分析データ（メトリクス履歴）との比較結果が分析に含まれていること。
- [ ] ユーザー承認フローを経て、実際に設定ファイルが更新されること。

# Requirements: Issue #26 - Issue Prioritization Automation

## Background
現在、プロジェクト内のIssueが増加しており、開発者が「次に取り組むべき最適なタスク」を即座に判断することが困難になっている。
単に古い順やランダムに選ぶのではなく、プロジェクト全体の進捗状況（project_state.md）や戦略的目標、および既存のIssue間の依存関係を考慮した、戦略的な優先順位付けを自動で提案する仕組みが必要である。

## User Story
開発者はスラッシュコマンド（例: `/next-issue`）を実行するだけで、エージェントによる現状分析と、次に着手すべきIssueの推奨案を受け取ることができる。

1.  **起動**: 開発者が `.gemini/commands/next-issue.toml` に定義されたコマンドを呼び出す。
2.  **分析**: エージェントが `project_state.md`（全体進捗）と `gh issue list`（未完了タスク）をスキャンする。
3.  **推論**: 依存関係、目標への寄与度、緊急度に基づいて優先順位を判断する。
4.  **提示**: エージェントが選定理由（Reasoning）と共に、候補となるIssue（あるいは1つに絞った推奨案）を提示する。

## Functional Requirements
- **Command Entry**: `.gemini/commands/` 下に専用のコマンド定義ファイルを作成する。
- **Context Analysis**: 以下の情報を入力として読み取る。
  - `project_state.md`: プロジェクトの全体目標と現在のフェーズ、マイルストーン。
  - `gh issue list`: 未完了のIssue一覧（タイトル、ラベル、更新日時など）。
  - 必要に応じてIssueの本文や既存の設計ドキュメント。
- **Decision Logic**: 
  - プロジェクトのボトルネック解消を優先するロジック。
  - 技術的負債の解消と新機能開発のバランス。
  - Issue間の「ブロック（依存）」関係の考慮。
- **Output**: 
  - 分析結果（現状の要約と、なぜそのタスクが今必要なのかの根拠）。
  - 推奨されるIssue IDとその概要。
  - 次のアクション（Deep Diveの開始など）の具体的な提示。

## Success Criteria
1.  専用のスラッシュコマンド（例: `/next-issue`）が正常に定義され、実行可能であること。
2.  分析プロセスにおいて、単なるリスト表示ではなく「なぜこれなのか」という納得感のある選定理由が提示されること。
3.  提示されたIssueに対して、開発者が迷うことなく次のステップ（実装や詳細設計）に進めること。

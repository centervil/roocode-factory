# Requirements: Issue #19 外部依存関係のリファクタリング

## 1. 背景
現在、`my-skills` 内の各スキルは、特定のプロジェクト構造（例: `AGENTS.md`、`.ops/` ディレクトリ、絶対パス）に強く結合しています。このため、別のプロジェクトにコピーして使用しようとするとパスエラーが発生します。Phase 4 (Release Candidate) の目標である「ポータビリティの確保」を達成するため、これらの依存関係を排除し、動的にプロジェクト環境を認識できるようにリファクタリングする必要があります。

## 2. ゴール & ユーザーストーリー
- **ゴール**: `my-skills` ディレクトリを新しいプロジェクトにコピーするだけで各スキルが動作するか、あるいは最小限の設定（Issue #23 の初期化スキル）で動作可能な基盤を整える。
- **ユーザーストーリー**: 
  - 新しいプロジェクトを開始する開発者が `my-skills` を導入した際、`skill-requirement-expert` や `skill-project-and-skill-architect` が現在のディレクトリから `AGENTS.md` を自動的に見つけ出し、即座に分析や設計を開始できる。

## 3. 詳細要件
- **R1: 依存関係の調査 (Audit)**: `work/my-skills/` 内の全ファイル（.md, .py, .sh 等）を対象に、プロジェクト固有のパス（例: `/home/centervil/...`、`AGENTS.md`、`.ops/project_state.md`）へのハードコードされた参照を全数調査する。
- **R2: 動的なパス解決**: 
  - `AGENTS.md` や `README.md` などの主要ファイルを参照する際、カレントディレクトリ（CWD）から上位ディレクトリへ遡ってファイルを探索する「自動探索ロジック」を優先的に採用する。
  - 自動探索が困難な場合は、環境変数（例: `ROOCODE_PROJECT_ROOT`）や共通の設定ファイルからパスを注入できる構造に変更する。
- **R3: ポータビリティを重視した設計**: 
  - スキル定義ファイル（`SKILL.md`）内の指示（Instructions）から特定のプロジェクト名や固有のパス名を排除し、一般的な用語（例: "Project Root", "Core Identity File"）に抽象化する。

## 4. 完了定義 (Acceptance Criteria)
- [ ] **調査レポートの作成**: `work/my-skills/` 内の依存関係調査結果リストが文書化されていること。
- [ ] **ハードコードされたパスの排除**: `grep` 検索において、特定のユーザー名（`centervil`）や絶対パスが `my-skills` 配下のファイルから検出されないこと。
- [ ] **動的探索の検証**: `AGENTS.md` が存在しない空のディレクトリでスキルを実行しても、パスエラーで異常終了せず、「ファイルが見つかりません」といった適切なメッセージが表示されること。
- [ ] **ポータビリティテスト**: `my-skills` を一時ディレクトリにコピーし、その場所で基本的なスキルの読み込みが可能であることを確認すること。

## 5. スコープ内外
- **スコープ内**: `work/my-skills/` 配下のすべてのスキルおよびスクリプトのリファクタリング。
- **スコープ外**: `README.md` や `AGENTS.md` 自体を自動生成するロジック（これは Issue #23 の範囲）。
